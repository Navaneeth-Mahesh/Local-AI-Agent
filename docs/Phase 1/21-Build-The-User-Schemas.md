# Why Start With Schemas?

Professional development usually follows this order:

```
Design API

↓

Create Schemas

↓

Business Logic

↓

Database

↓

Routes
```

Why?

Because the API contract comes first.

The frontend needs to know:

* What data to send
* What data it receives

Schemas define that contract.

---

# What is a Schema?

A schema is **NOT** a database model.

This confuses almost everyone.

Let's compare.

---

## SQLAlchemy Model

Represents:

```
Database Table
```

Example:

```
users

id

username

email

password_hash

created_at

updated_at
```

The database needs everything.

---

## Pydantic Schema

Represents:

```
HTTP Request

or

HTTP Response
```

It only contains the fields that the API should accept or return.

---

Think of it like this:

```
Browser

↓

Schema

↓

Service

↓

Repository

↓

Database Model

↓

Database
```

---

# User Registration

When someone registers,

they send:

```json
{
    "username":"Navaneeth",
    "email":"nav@gmail.com",
    "password":"MyPassword123"
}
```

Notice:

They do **NOT** send

```
id

created_at

updated_at

password_hash
```

Those are generated by the backend.

---

# Response

After registration,

we return:

```json
{
    "id":1,
    "username":"Navaneeth",
    "email":"nav@gmail.com"
}
```

Again,

we do NOT return

```
password

password_hash
```

# Code

Let's write it step by step.

---

## Imports

```python
from pydantic import BaseModel, ConfigDict, EmailStr
```

---

### Why BaseModel?

Every schema inherits from:

```python
BaseModel
```

It provides:

* Validation
* Serialization
* Type checking
* Automatic documentation

---

### Why EmailStr?

Instead of:

```python
email: str
```

we write:

```python
email: EmailStr
```

Now this:

```
abc@gmail.com
```

passes.

But

```
abc
```

fails automatically.

---

# Registration Schema

```python
class UserRegister(BaseModel):
    username: str
    email: EmailStr
    password: str
```

This is used by:

```
POST /auth/register
```

---

# Login Schema

```python
class UserLogin(BaseModel):
    email: EmailStr
    password: str
```

Used by:

```
POST /auth/login
```

Notice:

No username needed.

---

# User Response

```python
class UserResponse(BaseModel):
    id: int
    username: str
    email: EmailStr

    model_config = ConfigDict(
        from_attributes=True
    )
```

---

## What is `from_attributes=True`?

This is one of the most important Pydantic v2 features.

Suppose SQLAlchemy returns:

```python
user = User(...)
```

That's a Python object.

FastAPI needs JSON.

Normally,

Pydantic expects:

```python
{
    "id":1,
    "username":"Nav"
}
```

But SQLAlchemy gives:

```python
user.id

user.username
```

`from_attributes=True`

tells Pydantic:

> "Read values directly from object attributes."

Without it,

serialization fails.

---

# Token Schema

When login succeeds,

we return:

```json
{
    "access_token":"...",
    "token_type":"bearer"
}
```

Schema:

```python
class Token(BaseModel):
    access_token: str
    token_type: str
```

---

# Token Payload

When we decode JWT,

we expect:

```json
{
    "sub":"1"
}
```

Schema:

```python
class TokenPayload(BaseModel):
    sub: str
```

Later we'll extend it.

---

# Final File

Your completed `app/schemas/user.py` should look like this:

```python
from pydantic import BaseModel, ConfigDict, EmailStr


class UserRegister(BaseModel):
    username: str
    email: EmailStr
    password: str


class UserLogin(BaseModel):
    email: EmailStr
    password: str


class UserResponse(BaseModel):
    id: int
    username: str
    email: EmailStr

    model_config = ConfigDict(
        from_attributes=True
    )


class Token(BaseModel):
    access_token: str
    token_type: str


class TokenPayload(BaseModel):
    sub: str
```

# Why We Didn't Add Validation Yet

You might be wondering:

> Why isn't the password validated?

For example:

```python
password: str
```

accepts:

```
123
```

That's intentional.

We first build the architecture.

Later, we'll improve it with:

* Minimum length
* Maximum length
* Regex
* Strong password validation
* Username rules

This keeps each lesson focused.
